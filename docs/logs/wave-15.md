  이라 기능별로 쪼개면 3레인이 셸을 두고 충돌한다고 나왔다. → **파일 비겹침으로 재분할**: 레인1(셸·store·CSS 단독
  writer)·레인2(DetailOverlay·InspectorSpecPanels)·레인3(SpecView·icons·LeftRail·카피). 실제로 3레인 소스 겹침 0.

  얹는 게 안전하다는 판단 — 창업자에게도 "뼈대 먼저"로 합의.

## 핵심 설계 결정과 근거
- **상세 = 플로팅 전담(비모달), 도킹은 테이블 전용.** 레인1은 "우패널 숨김 + 선택 시 우패널 펴기"로, 레인2는
  "선택 시 플로팅 오픈"으로 각자 상세를 처리했다 — **파일은 안 겹쳤지만 "선택 시 상세 어디 뜨나"가 의미 충돌**.
  창업자 의도(#7 "클로드 플랜처럼 떠서 뒤도 조작")에 맞춰 **플로팅으로 일원화**: ① selectSpec*의 우패널 펴기 제거
  ② OverlayPanel에 **비모달 모드**(modal=false — aria-modal·백드롭 딤·포인터 차단·스크롤 잠금·포커스 트랩·자동
  포커스 전부 해제, Esc·닫기만 유지) 신설 ③ RightPanel은 명세 상세를 아예 안 그리게(테이블 전용) — 숨은 도킹
  인스펙터가 플로팅과 **DOM 중복**을 만들어 셀렉터가 다 모호해지던 걸 뿌리 제거.
- **자동 오픈 = 클릭 카운터(nonce).** 처음엔 "선택 id 변화 + sticky inspected"로 판정했으나, 크로스체크가
  "닫은 뒤 필터 보정으로 선택 id가 바뀌면 창이 스스로 재오픈"하는 결함을 잡았다. → store에 **specSelectClickSeq**
  (selectSpec* 클릭에만 증가, syncSpecSelection 비클릭은 미증가)를 두고 그 증가에만 오픈. 같은 항목 재클릭 재오픈도
  자연히 됨. 순수 함수(shouldAutoOpenDetail)라 nonce 비교 한 줄로 단순화.
- **프롬프트 = 좌측 리사이즈 도킹.** ChatDock의 챗 코어(useEditorChat·전송·계획·suggestions)는 위치 독립이라
  그대로 감싸고 하단 바 셸만 좌측 세로 패널로 교체. useResizable(280~400)은 순수 계산 분리 + 포인터/키보드 배선.
- **개수 제거는 aria-label까지.** 창업자 "개수 무의미"(#4) — 시각 span(레인3) + aria-label(통합) 둘 다 제거.
  결과로 apis/dbTables가 LeftRail에서 미사용 → prop 축소. 개수 뱃지에 의존하던 e2e는 **실물 검증으로 대체**
  (journey: API 개수 뱃지 → DataView에 upsert된 엔드포인트 존재; 그래프 반영 → 실 PATCH 200 + 토스트).

  **테스트를 동작에 맞춰 고친 것이지, 통과시키려 약화한 게 아니다** — 각 정정이 새 UX의 올바른 단언인지 확인.

## 배운 것 (프로세스)
- **파일 비겹침 ≠ 충돌 없음.** 레인1·2가 파일은 안 겹쳤는데 "선택 시 상세 어디"라는 **동작 계약**이 충돌했다.
  다음엔 파일 소유뿐 아니라 **공유 상태의 동작 계약(누가 무엇을 여닫나)**도 패킷에 동결해야.
- **"비차단"이라 적힌 프리미티브도 실제 구현을 확인하라.** OverlayPanel 주석은 "비차단 참조 창"인데 실제는
  aria-modal·백드롭·스크롤잠금 = 차단형이었다. 레인2가 evidence-first로 잡아 정직 신고 → 비모달 모드로 해소.
- **통합 e2e는 전체.** 서브셋만 돌리면 라벨 리네임 같은 크로스 스펙 회귀가 샌다(14차 policy-docs).
- **레인 정지 시 조기 오케스트레이터 마무리.** 레인3이 편집 완료 후 검증·커밋 전 ~50분 정지(반복 패턴) →
  미커밋 실물 검증 후 오케스트레이터가 게이트 재검증·커밋으로 살림. 워처 킬 빈발과 겹쳐 감지 지연.
