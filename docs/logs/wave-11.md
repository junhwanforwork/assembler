## 편성 판단
- **왜 이 웨이브였나:** 원래 12차(아크2-A) 계획이던 레포 연동을 사용자가 "이거 먼저 해결"로 당김.
  근거 = 실사용 마찰 실증(수동 JSON 흐름을 직접 겪음). UI/UX 조립은 12차로 밀림.
- **3분해 기준:** 추출 엔진(순수 lib) / 깃 클론 라우트(서버) / 연결 UI(FE) — 소유 완전 배타.
  병행 가능하게 **계약(시그니처)을 패킷에 동결**하고 소비 레인은 스텁으로 개발 → 통합에서 실물 결합.

## 핵심 설계 결정과 근거
1. **추출기 = 브라우저·서버 겸용 순수 함수** — 폴더 경로는 클라에서 돌려 서버 신규 표면 최소화,
   깃 경로만 서버(클론 필요). 같은 엔진이라 결과 일관.
2. **rate limit 신규 키 대신 sync 재사용** — RPC 인자 가드 fail-open 함정(CLAUDE.md) 회피, 마이그레이션 0.
3. **summary는 빈 문자열** — 코드에서 확인 안 된 설명을 지어내지 않는다(evidence-first). 설명은 12차 API 해석 AI 몫.
4. **차단 목록 = 읽기 전 필터**(브라우저에서도 text() 미호출) — "안전하게"의 경계를 UI 문구가 아니라 코드로.
