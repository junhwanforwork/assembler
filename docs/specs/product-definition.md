## 1. 제품 정의

**한 문장:** Assembler는 **비개발자가 더 안전하게 개발에 접근하는 행위**를 만든다 — 레포의 기획 문서(md)와 코드 실체(API·DB)를 **하나의 모델**로 연결해, 비개발자는 평문으로 기획하고, 개발자는 실체가 연결된 문서와 개선 추천을 받는다.

**해결하는 문제 — 기획과 개발 사이의 간극:**
- 비개발자는 레포를 두려워한다 — 뭘 건드리면 안 되는지 모르고, API·DB가 무슨 뜻인지 모른다.
- 개발자는 기획 문서를 불신한다 — 코드 실체와 안 맞고, 바뀌어도 전달이 안 된다.
- 기존 도구(노션·manyfast)는 **문서에서 끝난다** — 문서 속 "API"라는 단어와 실제 코드의 API가 남남이다.

**우리의 답 (해자):** 문서 속 모든 API·테이블·기능이 **실체(코드-진실)와 연결**되고, 클릭하면 **비개발자 말로 해석**된다. 스펙은 한 곳(모델), 변경은 한 번 — 문서 N종은 전부 그 모델의 절단면이다.

**이번에 하지 않는 것 (명시적 후퇴·컷):**
- 와이어프레임·UI 요소 — later phase (화면 숨김+생성 중단, 데이터 구조는 보존)
- 아이디어만으로 API 목록 생성(구 ASM-050) — 컷. 해석은 **실체가 있는 것**에만.
- env·시크릿·코드 파일 연동 — md 문서만. "안전하게"의 경계선.

## 2. 사용자와 핵심 여정

**주 사용자 = 비개발자** (기획자·창업자·바이브 코더): 레포가 있거나(팀·바이브 코딩 산출물) 아이디어만 있다.
**부 사용자 = 개발자**: 문서를 받아 구현하고, 역으로 추천을 준다.

**여정 A — 비개발자 (핵심):**
1. **연결**: 레포의 기획 md·API·DB를 가져온다 (또는 아이디어로 시작해 생성)
2. **이해**: 문서·데이터 사전에서 API·테이블을 비개발자 말로 해석받는다 ("이 테이블은 주문을 담아요, 결제 테이블과 연결돼요")
3. **기획**: 평문으로 요구사항·기능을 쓰고 고친다 (챗 변경 → 한글 diff 승인 → 영향 범위 확인)
4. **작성**: 정책 문서(계정 구조·요금 등)를 직접 쓴다
5. **전달**: md로 저장 → git에 올린다(PR) → 개발자가 받는다

**여정 B — 개발자:**
1. 전달받은 문서·플로우를 본다 (기술 명세 — 기능→API·DB 연결이 코드 관점으로)
2. 구현 컨텍스트를 내보낸다 (코딩 에이전트용 패키징 — 기존 기능)
3. **추천을 받는다**: "이 화면에 이 API 개선이 필요해 보여요" — 문서·플로우 기반 제안

**북극성 지표 (유지):** 변경 1건당 수정 지점 수 = 1 · 변경 반영 소요 시간.

## 3. 기능 정의

### F1 · 시작 — 3경로
| 경로 | 정의 | 상태 |
|---|---|---|
| A 아이디어 | Composer에 아이디어 → 스펙 그래프 생성 (요구사항·기능·페이지·플로우 — **와이어 제외로 개정**) | 있음(생성 계약 개정 = 9차) |
| B 코드 연결 | API·DB 목록 싱크-인 (수동 JSON → 아크2에서 레포 자동) | 있음 |
| C 레포 md 연동 | 레포의 기획 문서(md)를 가져와 모델과 연결. **env·시크릿·코드 제외 — 문서 파일만** | 신규(아크 2) |

### F2 · 모델 (코어 — 모든 것의 원본)
- 스펙 그래프: `요구사항 → 기능 → 페이지 · 플로우` + **`기능 → API · DB`** (기능이 연결의 1급 주인 — feature.dbTableIds 추가로 비대칭 해소)
- API·DB = 코드-진실(읽기 전용, 싱크-인 유입) — 모델이 참조하되 지어내지 않는다
- 와이어프레임·요소 = 휴면 (타입·데이터 보존, 생성·표시 중단)
- 무결성: 끊어진 참조는 저장에서 거부(409) — 유지

### F3 · 문서 — 투사형 (모델의 절단면 · 저장 안 함 · 모델을 고치면 문서가 바뀜)
| 문서 | 독자 | 답하는 질문 | 내용 |
|---|---|---|---|
| PRD | 모두 | 왜·무엇을 | 요구사항→기능 구조 (있음) |
| **기술 명세** | 개발자 | 어떻게·뭘로 | 기능→API·DB 연결을 코드 관점으로, 본문 속 API·테이블에 해석 카드 인라인 |
| **데이터 사전** | 비개발자 | 데이터가 뭔데 | 테이블별 의미·관계·쓰임새를 평문으로 (db-learning 노트 승격) |

공통 문법(클로드식 문서): 목적별 종류 · 고정 구조 · **본문 속 실체 인라인 링크** · 모델이 바뀌면 문서도 바뀜(살아있음). 함정 방지: 문서끼리 복붙하는 병렬 문서 금지 — 전부 한 모델의 투사.

### F4 · 문서 — 작성형 (문서가 원본인 유일한 종류)
- **정책 문서**: 계정 구조·요금·권한 같은 정책을 에디터에서 직접 작성
- md로 저장 → git 업로드(PR) — 여정 A-5의 실체
- 스펙 그래프 참조는 링크로 (기능·API를 언급하면 연결 — 복붙 아님)

### F5 · 해석 (신 방향의 심장 — "안전하게"를 만드는 것)
- DB 해석: 테이블 카드에 AI 설명 (있음 — 구조=사실/설명=추론 분리, 환각 차단)
- **API 해석**: 엔드포인트에 같은 문법의 AI 설명 (신규 — v1.5)
- **문서 인라인 해석**: 문서 본문의 API·테이블 이름 위에서 바로 해석 카드 (신규 — F3와 결합)
- 원칙: 해석은 실체(코드-진실)에만. 추정은 "AI 추정" 표시.

### F6 · 변경 (있음 — 유지·강화)
- 챗 변경 계획 → 한글 diff 승인 → 적용 → 영향 범위 자동 표시 → 활동 기록
- 변경이 문서 3종에 즉시 반영되는 것("스펙은 한 곳, 변경은 한 번")이 데모의 핵심 장면

### F7 · 전달 (출구)
- 구현 컨텍스트 내보내기 (있음)
- **md·PR로 레포 반영** (신규 — F4와 결합, 아크 2)
- **개발자 추천**: 모델·플로우를 근거로 "이 화면에 이 작업" 제안 (신규 — 기존 suggestions 재정향, 유료 자동발사 제거 선행)

## 4. IA — 정보 구조

### 4-1. 화면 구조
```
대시보드
├─ 제품 목록 (제품 = API·DB 코드-진실의 소유 단위)
└─ 시작: 아이디어로 시작 · 코드 연결 · (아크2) 레포 연동

에디터 (제품/스펙 진입 후)
├─ 좌 레일
│   ├─ 제품 구조: 명세 · 문서 ▾(PRD·기술 명세·데이터 사전) · 유저플로우   ← 와이어 행 제거, 문서가 패밀리로
│   ├─ 정책 문서 (작성형 — 신규 그룹, 아크2)
│   └─ 공통: DB · API
├─ 중앙: 선택한 뷰 (명세/문서/플로우/데이터)
├─ 우 패널: 인스펙터 = 선택 대상의 상세 + 해석 카드
├─ 하단: 챗 도크 (변경 계획·승인)
├─ 오버레이: 문서 패널 — 열고 닫는 임시 창 (입체감, 클로드 데스크톱 스타일)  ← 신규 UX (v1.5)
└─ (아크2) git 상태 바: 레포 · 브랜치 · 변경량 +N/−N · "PR 생성" · CI 칩
```

### 4-2. 데이터 구조 (저장 층)
| 저장소 | 담는 것 | 변경 |
|---|---|---|
| asm_products (+asm_apis·asm_db_tables) | 제품 + API·DB 코드-진실 | 유지 |
| workspace.design (그래프 jsonb) | 요구사항·기능·페이지·플로우 (+휴면: 와이어·요소) | feature.dbTableIds 추가 |
| asm_db_table_notes | DB 해석 | 유지 → API 해석용 확장(v1.5) |
| **정책 문서 저장** | 작성형 md 문서 | 신규(아크2 — 테이블 설계 그때) |
| 투사 문서 | **저장 안 함** — 렌더 시 모델에서 계산 | 원칙 유지 |

### 4-3. 연결 지도 (누가 누굴 참조)
```
요구사항 ←─ 기능 ─→ 페이지 ←─ 플로우(엣지)
              │
              ├─→ API (코드-진실)        ← 기능이 연결의 1급 주인
              └─→ DB 테이블 (코드-진실)    ← 신규 필드 (feature.dbTableIds)
정책 문서(작성형) ─링크→ 기능·API·테이블 (복붙 금지)
투사 문서 3종 = 위 그래프의 절단면 (자체 데이터 0)
```

## 5. 단계 (phase)

| 단계 | 내용 | 웨이브 |
|---|---|---|
| **v1 마감** | 와이어 후퇴 + 생성 다이어트(속도) + 기능→DB 승격 + 문서 3종 투사 + 인라인 해석(기존 노트) + 유료 발사 제거 | 9차 |
| **v1.5** | 정책 문서 작성·md 저장(다운로드) + 레포 md 읽기 연동 + 오버레이 문서 패널 + API 해석 AI | 10차 |
| **v2** | git 쓰기(PR) + git 상태 바 + 개발자 추천 | 11차 |
| later | 와이어 부활 검토 · MCP · 협업 | M3 판정 후 |

각 단계 끝 = 사용자 직접 QA 가능 상태 유지. **1차 목표(재정의): "아이디어 또는 레포 → 기획 문서 3종 + 해석을 로컬에서 완주."**
