## 북극성 선언

> **기획 문서·API·데이터가 한 모델로 연결돼 있고 — 문서 속 모든 실체가 비개발자 말로 해석되며 —
> 제품이 v1.0에서 v1.1로 갈 때 무엇이 왜 바뀌고 어디에 영향 주는지 구조가 알려준다.**
>
> **— 스펙은 한 곳, 변경은 한 번. Assembler는 비개발자가 더 안전하게 개발에 접근하는 행위다.**

해결하는 고통(창업자 원문 기준): 피그마 인라인 버전, 제각각인 파일·페이지 기준, 수정 하나에 피그마+여러 문서를 전부 손으로 고치는 산탄 반영 — 그래서 v1.0→v1.1이 오래 걸린다. **+ (2026-07-07) 기획↔개발 간극**: 비개발자는 레포가 두렵고 API·DB가 무슨 뜻인지 모르며, 개발자는 코드 실체와 안 맞는 기획 문서를 불신한다.

## 위계 — 목적·수단·가속기

| 층 | 내용 | 근거 |
|---|---|---|
| **목적** | 변경이 싼 연결된 단일 스펙 | v1.0→v1.1 고통이 가장 구체적이고, 경쟁 도구가 구조적으로 못 하는 지점 |
| **수단** | 인과 연결 구조 — "사용자가 이걸 하면, 다음에 무엇이 일어나는가?" (**요구사항→기능→페이지·플로우, 기능→API·데이터** — 2026-07-07 개정: 화면 속 동작[와이어·요소] 마디는 later phase, 기능이 API·데이터 연결의 1급 주인) | 연결돼 있어야 변경 영향이 자동 추적되고 diff가 구조적이 됨. 이 사슬이 곧 기획자↔개발자 번역층 — 기획자는 사슬을 따라가면 API·데이터가 보이고(+**해석**으로 이해하고), 개발자는 거슬러 올라가면 의도가 보인다 |
| **가속기** | AI 생성·제안·리뷰 = "사용자에게 더 나은 선택지 제공" | 스펙을 빨리 만들고 빈틈 없이 유지하게 하는 엔진. 목적 아님 — 생성은 경쟁사도 한다 |

위계 규칙: 우선순위가 충돌하면 위층이 이긴다. 가속기를 위해 목적을 희생하지 않는다.

## 지표

- **북극성 지표: 변경 반영 소요 시간** — "바꾸자" 결정부터 스펙 전체가 정합하게 갱신·공유되기까지.
- **설계 원칙(선행 지표): 변경 1건당 수정 지점 수 = 1** — 한 곳 고치면 연결된 모든 곳에 반영된다. 새 기능 설계 시 이 원칙을 깨는 설계는 반려.

## 강등 목록 — 이 북극성이 정리한 파편들

| 파편 | 판정 |
|---|---|
| 단일 PRD export | 목적 아님 → **반출 포맷**. 목적은 "정합한 스펙 상태", export는 그 스냅샷 + diff |
| AI 생성 vs AI 리뷰 대립 (generate ↔ suggestions 프롬프트 철학 분열) | 해소 — 같은 렌즈("더 나은 선택지")의 국면 차이: 생성=첫 선택지, 제안=다음 선택지, 리뷰=놓친 선택지 |
| 팀 공유·협업 | 2차. 단 "결정의 버전·이력"은 핵심이므로 별도 취급 |
| Dev Mode(코드 컨텍스트·Git·Merge) | v1 제외 유지. 코드 git을 대체하지 않는다는 원칙 유지 — **스펙 자체의 버전 관리는 별개 개념**이며 핵심이다. (2026-07-07 보정: 아크 2의 git 연동은 **문서 파일만** — 코드 git 대체가 아니라 기획 md의 반출입 통로) |
| 와이어프레임·UI 요소 (2026-07-07) | **later phase 강등** — "아무리 가르쳐도 안 는다"(생성 품질), 비용·속도의 최대 소비처. 화면 숨김+생성 중단, 타입·데이터 보존(부활 비용 낮게). M3 판정이 지목할 때만 부활 |
| 아이디어→planned API 생성 (구 ASM-050) | **컷** — 해석은 실체(코드-진실)가 있는 것에만. 지어낸 API 목록은 "괜한 것" |

## 경쟁 포지셔닝 — manyfast.io (2026-07 확인 기준)

manyfast 현황: "대화하세요, 기획이 완성됩니다" — PRD·기능명세서·유저플로우·와이어프레임을 한 에디터에서 AI 대화로 생성, 엑셀·이미지·마크다운 내보내기, Cursor·Claude Code MCP 연동, 실시간 협업. 크레딧 과금.

⚠️ `builder-layout.md`의 "매니패스트는 단일 PRD 문서" 비교는 **낡았다**(문서 4종 + MCP 출시됨). 경쟁 사실관계는 이 문서를 기준으로 한다.

### 전장 판정

| 영역 | 판정 |
|---|---|
| AI 대화로 기획 생성 | ❌ 계약된 전장 — 테이블 스테이크. 여기서 포지셔닝하지 않는다 |
| 여러 산출물 한 공간 | ❌ 계약된 전장 |
| MCP 내보내기 (문서 서빙) | ❌ 계약된 전장 |
| **산출물 간 인과 연결** | ✅ 해자 — manyfast는 문서 4종의 *병렬* 배치, 서로 참조하는 구조 아님. 변경 시 여전히 4곳 수정(산탄 반영이 한 지붕 아래로 이사만 한 것) |
| **구조적 버전·diff** | ✅ 해자 — 버전 개념 자체가 없음 |
| **개발 언어 층 (API·데이터 번역층 + 해석)** | ✅ 해자 — manyfast의 문서는 실체와 남남(문서 속 "API"라는 단어와 실제 코드가 무관). 우리는 문서 속 API·테이블이 코드-진실과 연결되고 **비개발자 말로 해석**된다 (2026-07-07: 와이어 없이도 이 해자는 성립 — 오히려 순수해짐) |

### 프레임

> **manyfast = 0→1 (첫 초안을 빨리, "완성") / Assembler = 1→N (버전을 거듭할수록 가치가 커지는 연결된 스펙, "진화")**

대항 메시지: 기획은 "완성"되는 게 아니라 계속 바뀐다. 진짜 비용은 쓰는 날이 아니라 **고치는 날** 발생한다. — "스펙은 한 곳, 변경은 한 번. v1.1→v1.2의 diff가 보인다."

획득(acquisition)은 AI 생성이 담당하되(1→N만으로는 첫 세션 가치가 약함), 포지셔닝·메시지·로드맵 우선순위는 진화·버전에 건다.

## 실행 순서 원칙 — 기능 N개가 아니라, 모델 1개 + 투사 N개

모든 기능은 연결된 모델 **위에** 만든다. 문서가 원본이 되는 순간 산탄 반영이 재발한다.

**문서 2계층 (2026-07-07 확정):**
- **투사형 문서** (PRD·기술 명세·데이터 사전): 모델이 원본 — 문서는 렌더 시 계산되고 저장하지 않는다. 문서끼리 복붙하는 병렬 문서는 금지(manyfast 자기부정 함정).
- **작성형 문서** (정책 문서 — 계정 구조·요금·권한 등): **문서가 원본인 유일한 예외.** 직접 작성하고 md로 저장·git 업로드한다. 단 스펙 그래프(기능·API·테이블)를 언급할 땐 복붙이 아니라 **링크(참조)** — 원칙 위반이 아니라 원칙의 경계 명시다.

| 순서 | 무엇 | 상태 |
|---|---|---|
| ① | 연결 모델 + AI 생성 | 있음 (`assembler-generate.ts`가 문서가 아니라 연결 구조를 생성) — 유지·다이어트(와이어 생성 중단) |
| ② | 투사 뷰 — PRD·기술 명세·데이터 사전·플로우는 모델을 다른 각도로 렌더한 것 + **해석 인라인**(문서 속 API·테이블에 비개발자 설명) | 문서 패밀리 확장 중(9차). "문서 편집"이 아니라 "모델 편집의 다른 각도" |
| ③ | 스펙 버전 + 구조적 diff | Recent Activity 타임라인이 씨앗. **v1 편입 필요(후속 결정)** |
| ④ | 레포 연동 — 기획 md 읽기(env·시크릿·코드 제외)·정책 문서 쓰기(PR)·개발자 추천 | 아크 2 (③과 독립 착수 가능 — 문서만 다루므로) |
| ⑤ | MCP (diff-aware) — 질의 가능한 스펙 + diff 컨텍스트 | ③ 이후에만. 버전 없는 MCP는 카피 |
