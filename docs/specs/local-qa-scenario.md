# 로컬 기능 QA 각본 — 1차 목표 완주 (ASM-051)

> 이 각본은 **사용자가 직접** 로컬에서 제품을 처음부터 끝까지 써 보고,
> "1차 목표(문서 3종 + 해석 로컬 완주)"가 진짜 되는지 확인하는 체크리스트예요.
> 11차 웨이브 **완성 후 상태** 기준이에요(문서 3종 · 와이어프레임 숨김 · 코드 연결 3경로).
> 정책 문서 작성은 다음 단계 기능이라 이 각본에 없어요.

---

## 시작 전 준비

- [ ] 터미널에서 `npm run dev`를 실행해요.
- [ ] 브라우저에서 `http://localhost:3000`을 열어요.
- [ ] **"무엇을 만들어 볼까요?"** 라는 큰 제목이 보이면 준비 완료예요.

**비용 안내:** 이 각본은 AI 유료 호출이 **3번쯤** 나가요
(A-1 생성 1번 · A-5 챗 변경 1번 · B-3 설명 생성 1번). 그 외 단계에서는 호출이 없어야 정상이에요.

**기록 방법:** 각 단계의 "이게 보이면 정상"과 다르면, 맨 아래 **실패 기록표**에 적어 주세요.
시간을 재는 단계(⏱ 표시)는 휴대폰 스톱워치면 충분해요.

---

## 시나리오 A — 아이디어로 시작해서 내보내기까지

### A-1. 아이디어 입력 → 생성 ⏱

- [ ] 가운데 입력창("만들고 싶은 제품이나 기능 아이디어를 적어보세요…")에
      만들고 싶은 것을 2~3문장으로 적어요. (예: "반려견 산책 기록 앱. 산책을 시작·종료로 기록하고, 주간 리포트를 보여줘.")
- [ ] ⏱ **스톱워치 시작** → 보내기 화살표 버튼을 눌러요.
- [ ] "구조를 만들고 있어요…" 문구가 보여요.
- [ ] 끝나면 **에디터 화면으로 자동 이동**해요. ⏱ **여기서 스톱워치 정지 → 아래 지표 기록란에 "생성 소요 시간" 기록.**

**이게 보이면 정상:** 에디터 왼쪽에 "제품 구조" 목록이 보이고, 각 행에 1 이상의 숫자가 붙어 있어요.

### A-2. 구조 훑어보기

- [ ] 왼쪽 "제품 구조"에 **문서 · 기능명세서 · 유저플로우** 행이 보여요.
- [ ] **"와이어프레임" 행은 보이지 않아야 정상이에요.** (이번에 화면에서 뺐어요 — 보이면 실패로 기록)
- [ ] 기능명세서를 눌러 기능 목록이 나오는지 확인해요. 기능을 하나 누르면 오른쪽 패널에 상세가 떠요.

### A-3. 문서 3종 확인

- [ ] 왼쪽에서 **문서**를 눌러요. 문서 종류를 고를 수 있어요: **PRD · 기술 명세 · 데이터 사전**.
- [ ] **PRD**: 요구사항과 기능이 이어진 구조가 글로 보여요.
- [ ] **기술 명세**: 기능마다 API·DB 연결이 보이는 문서예요. 지금은 코드를 연결하기 전이라
      **"연결된 API·테이블이 없다"는 안내가 나오는 게 정상**이에요.
      ⚠️ 여기서 **지어낸 API나 테이블 이름이 보이면 실패**로 기록해 주세요(없는 걸 있다고 하면 안 돼요).
- [ ] **데이터 사전**: 마찬가지로 아직 테이블이 없으니 빈 상태 안내가 나오는 게 정상이에요.

### A-4. 유저플로우 확인

- [ ] 왼쪽에서 **유저플로우**를 눌러요.
- [ ] 화면(페이지) 카드들이 연결선으로 이어진 그림이 보여요.

### A-5. 챗으로 변경 1건 — 승인 → 적용 → 영향 확인 ⏱ (북극성 지표)

- [ ] 아래쪽 챗 입력창을 **클릭만** 해 봐요. 챗 영역이 펼쳐지되,
      **"추천을 불러오고 있어요…" 같은 추천 문구가 저절로 뜨지 않아야 정상**이에요
      (클릭만으로 유료 호출이 나가던 문제를 없앴어요).
- [ ] ⏱ **스톱워치 시작** → 변경을 한 문장으로 요청하고 Enter.
      (예: "결제 기능을 추가해줘")
- [ ] 잠시 후 **변경 계획 카드**가 떠요 — 무엇을 추가/수정/삭제하는지 **한국어로** 보여요.
- [ ] "적용하면 연결된 이 객체들까지 영향이 닿아요." 라는 **영향 범위**가 보여요(연결된 기능·요구사항 칩).
- [ ] **적용하기**를 눌러요 → "변경 계획을 스펙에 반영했어요." 가 떠요.
- [ ] 왼쪽 행의 숫자가 늘었는지 확인해요(예: 문서의 요구사항 수 +1).
- [ ] **문서 → PRD**를 다시 열어 방금 추가한 내용이 문서에도 들어와 있는지 확인해요.
      ⏱ **확인한 순간 스톱워치 정지 → "변경 반영 소요 시간" 기록.**
- [ ] **수정 지점 수 세기:** 이 변경 1건을 위해 내가 직접 고친 곳이 몇 군데였나요?
      챗 요청 1번 말고 다른 화면을 손댔다면 그 횟수를 적어 주세요. **목표는 1(챗 요청 한 번)이에요.**

### A-6. 내보내기

- [ ] 화면 위쪽의 **내보내기**를 눌러요.
- [ ] "기능 선택"에서 기능을 하나 이상 체크해요 → 미리보기에 연결된 명세가 나와요.
- [ ] 복사를 누르면 "복사했어요", 파일로 받기를 누르면 "파일로 받았어요"가 떠요.
- [ ] 미리보기 안에 방금 A-5에서 바꾼 내용이 들어 있는지 확인해요.

## 시나리오 B — 코드 연결(폴더 선택 또는 깃 주소) → 해석 확인

### B-1. 대시보드로 돌아가기

- [ ] 화면 왼쪽 위 로고(대시보드로 돌아가기)를 눌러요.

### B-2. 코드 연결하기 — 폴더 선택 또는 깃 주소

- [ ] **"이미 코드가 있어요"** 버튼을 눌러요.
- [ ] 다음 둘 중 편한 방법으로 연결해요:
  - **내 폴더 선택하기**(기본): 컴퓨터에 있는 프로젝트 폴더를 골라요. 잠시 후 미리보기가 떠요.
  - **깃 주소**: 공개 GitHub/GitLab 주소를 넣고 **가져오기**를 눌러요 → "레포를 읽고 있어요…" 후 미리보기가 떠요.
- [ ] 미리보기에서 확인해요:
  - **"API n개 · 테이블 m개를 찾았어요"**가 떠요.
  - `.env` 같은 민감한 파일이 폴더에 있었다면 **"n개 파일은 안전을 위해 읽지 않았어요"**가 떠요(눌러서 파일명 확인 가능).
  - API·테이블이 없는 폴더면 **"찾지 못했어요"** 안내가 뜨고 연결 버튼이 없어요 — 빈 연결은 안 돼요.
- [ ] **연결하기**를 눌러요 → **"코드를 연결했어요"** (또는 "…메인 스펙을 만들었어요") 알림이 떠요.
- [ ] 연결까지 마쳤으면 **B-2′는 건너뛰고 B-3로** 가요.

### B-2′. (개발자용 대안) JSON 직접 넣기

폴더·깃 주소를 쓸 수 없을 때만 사용해요. 아래 JSON이면 B-3 이후 확인 항목을 그대로 재현할 수 있어요.

- [ ] 모달 아래 **"JSON 직접 넣기 — 개발자용"**을 펼쳐요.
- [ ] 아래 JSON을 그대로 복사해서 "코드 정보 JSON" 칸에 붙여넣어요:

```json
{
  "apis": [
    { "method": "GET", "endpoint": "/walks", "summary": "산책 목록 조회", "status": "active", "source": "code" },
    { "method": "POST", "endpoint": "/walks", "summary": "산책 기록 저장", "status": "active", "source": "code" }
  ],
  "tables": [
    { "name": "walks", "description": "산책 기록", "columns": [
      { "name": "id", "type": "uuid", "nullable": false, "isPrimaryKey": true },
      { "name": "dog_id", "type": "uuid", "nullable": false, "isPrimaryKey": false },
      { "name": "started_at", "type": "timestamp", "nullable": false, "isPrimaryKey": false }
    ], "source": "code" },
    { "name": "dogs", "description": "반려견 정보", "columns": [
      { "name": "id", "type": "uuid", "nullable": false, "isPrimaryKey": true },
      { "name": "name", "type": "text", "nullable": false, "isPrimaryKey": false }
    ], "source": "code" }
  ]
}
```

- [ ] 연결 버튼을 눌러요 → **"코드를 연결했어요"** (또는 "…메인 스펙을 만들었어요") 알림이 떠요.

### B-3. DB 화면 + AI 해석 확인 (유료 1건)

> 폴더·깃 주소로 연결했다면 아래 `walks`·`dogs` 자리는 **내 코드에서 찾은 테이블·API 이름**으로 바꿔 읽어요.

- [ ] 에디터로 들어가 왼쪽에서 **DB**를 눌러요.
- [ ] 테이블 목록에 `walks` · `dogs`가 보이고, "무엇을 담는 곳" · "주요 컬럼" · 출처가 표로 보여요.
- [ ] `walks` 테이블을 눌러요 → 오른쪽 패널에 컬럼 구조(사실)가 먼저 보여요.
- [ ] 그 아래 **"AI 설명"** 섹션에 **"AI 추정"** 배지가 붙어 있어요 — 사실과 추정이 구분돼야 정상이에요.
- [ ] **설명 생성하기**를 눌러요 → 잠시 후 이 테이블이 무엇을 담고 어디에 쓰이는지 **쉬운 말 설명**이 나와요.
- [ ] 설명 속 내용이 붙여넣은 컬럼·연결과 맞는지 확인해요. ⚠️ **JSON에 없는 컬럼이나 테이블을 언급하면 실패**로 기록해 주세요.

### B-4. API 화면 확인

- [ ] 왼쪽에서 **API**를 눌러요.
- [ ] `GET /walks` · `POST /walks`가 "무엇을 하나요" 설명과 함께 표로 보여요.

### B-5. 문서에서 실체 확인

- [ ] 왼쪽에서 **문서 → 데이터 사전**을 열어요.
- [ ] 연결한 테이블(`walks` · `dogs`)이 문서 본문에 나오고, 테이블 이름에서 **해석(설명) 카드**를 바로 볼 수 있어요.
- [ ] **기술 명세**를 열어요. 기능과 API·DB의 연결이 아직 없다면 **"연결 없음"을 정직하게** 보여줘야 정상이에요
      (지어낸 연결이 보이면 실패로 기록).
