# Assembler 로드맵 마일스톤 — 웨이브의 종착점과 중단 규칙

> 확정: 2026-07-02 · "웨이브를 돌려 결국 무엇을 얻는가"에 대한 답의 단일 출처.
> 지표 정의는 [north-star.md](north-star.md)가 원본 — 여기 복제하지 않는다(스펙은 한 곳).
> 첫 검증 방식 = **본인 도그푸딩** (2026-07-02 사용자 확정).

---

## 왜 이 문서인가

웨이브가 "백로그 소진"을 목표로 돌면 끝나지 않는다 — 매 웨이브의 크로스체크·리뷰가 새 티켓을
낳기 때문에(ASM-027·028이 그 산물) 백로그는 자기 증식한다. 웨이브의 사유는 백로그가 아니라
**마일스톤 탈출 조건**이어야 한다. 이 문서가 그 탈출 조건과 중단 규칙을 정의한다.

---

## 마일스톤

### M1 · 경험 완성 — 기능 완결 → 점검 → 갭 마감 → 디자인 고도화 → 실데이터 검증

> 2026-07-03 개정: 사용자 3단계 제안(기능 점검 / 디자인 고도화 / 전체 재점검+테스트)을
> M1의 하위 단계 A~E로 편입. 순차 진행 — 각 단계의 산출이 다음 단계의 입력이다.

#### M1-A · 루프 완결 (= 5차 웨이브, 패킷 발급 완료)
- **ASM-029** 변경 전파 시각화 — 영향 범위를 구조가 알려주는 유일 차별 자산
- **ASM-030** 내보내기 모달 — 연결을 살린 구현 컨텍스트로 루프를 닫는 출구
- **ASM-028·027** 배포 전 방어(rate limit·잔여 하드닝) — M2의 선행 조건
- 탈출: 아이디어→생성→편집→AI 변경(≤3 인터랙션)→영향 확인→내보내기 완주 e2e 고정.

#### M1-B · 기능 총점검 (ASM-032)
1. **assembler 스펙을 assembler에 시드**(로컬) — v1-spec·north-star의 요구사항→기능→페이지→
   API·DB를 Composer/싱크-인으로 입력. 점검용 실데이터 + 디자인 평가용 진짜 콘텐츠 +
   M2 도그푸딩 데이터를 겸한다.
2. 여정×각도 점검 매트릭스 — 온보딩 B·C 경로 × 5각도 × 편집·챗 변경·전파·내보내기·기록.
   체크리스트 원본 = editor-interactions.md. ux-audit 낡은 항목(3~5차로 해소분) 닫기 포함.
3. 탈출: 갭 리포트 산출. **CRITICAL/HIGH만 M1-C 편입**, 나머지는 M3 판정 대기(중단 규칙 2).

#### M1-C · 갭 마감 웨이브 (6차)
- **ASM-033** DocView 읽기 투사 — PRD 투사 렌더(편집 없음, north-star "투사 뷰" 원칙)
- **ASM-034** WireframeView 읽기 구조 렌더 — UIElement 구조(#46 계약 소비,
  flow-view-pattern 재사용, 실렌더·편집 아님)
- + M1-B 발견 CRITICAL/HIGH
- 탈출: 5각도 전부 "준비 중" 표면 0 — 시작부터 끝까지 빈 각도 없이 경험 가능.

#### M1-D · 디자인 고도화 (스코프 폭발 방지 구조)
1. **ASM-035** 기계 부채 소탕(자명, 진단 불필요) — TYPOGRAPHY 토큰 CSS var 노출
   (globals.css←design-tokens.ts 1:1) → font-size 하드코딩 ~100건 토큰 치환 → 세그먼트
   컨트롤 1벌 프리미티브화 → Tooltip/Chip 소비·스피너 통합·죽은 챗 CSS 삭제.
2. **ASM-036** 인지 진단→선별 실행 — ui-ux-designer + assembler-design 병렬 진단.
   기준 = ux-references.md 3사 + 확정 디자인 방향(중성 블랙+블루·플로팅 라운드·그라데이션
   금지) + 정보 위계·시선 흐름·인지 부하. 우선순위 리포트 → **사용자 승인 top-N만 실행.**
- 탈출: font-size 하드코딩 0 · 세그 문법 1벌 · 진단 HIGH 해소 · 사용자 시각 승인 1회.

#### M1-E · 전체 재점검 + 실데이터 테스트 (ASM-037)
- 시드 스크립트 신설(가상 제품 + M1-B assembler 스펙 픽스처화, x-session-id RLS 정합).
- **실 DB 여정 e2e 1벌 신설** — 생성만 픽스처 주입(AI 호출 0 원칙 유지), 저장·PATCH·전파·
  내보내기·싱크-인은 모킹 없이 실 dev+DB 관통. 기존 모킹 e2e는 계약 검증용으로 유지.
- /health 진단 스윕 + 회귀 비교로 마감.

**M1 최종 탈출 조건:** 실 DB 여정 e2e 통과 + 시작→끝 경험에 빈 각도·거짓 표면 0 +
M1-D 탈출 조건 충족.

### M2 · 검증 가능 상태 — 배포 + 계측 (도그푸딩용 최소)

- **ASM-011** Vercel 배포 — 함수 리전 icn1(서울, Supabase ap-northeast-2 정합) · env ·
  배포 게이트(tsc·lint·build·e2e). 도그푸딩용이므로 공유·온보딩 고도화는 하지 않는다.
- **ASM-031** 도그푸딩 시나리오·지표 기록 — **텔레메트리 인프라를 만들지 않는다.** 수동 측정:
  시나리오 문서(예: "기능 1개 추가 + 변경 3건 반영")를 배포 URL에서 수행하며 북극성 지표
  (변경 반영 소요 시간 · 변경 1건당 수정 지점 수)를 직접 기록. asm_activity 델타(ASM-019)가
  보조 증거.

**탈출 조건:** 배포 URL에서 시나리오 1회 완주 + 지표 기록 1세트.

### M3 · 판정 — 빌드 중단 게이트

- 도그푸딩 실사용 ≈1주 → **판정 세션**: 지표·마찰 로그를 근거로 다음 아크를 결정한다.
- 도그푸딩 중 발견한 마찰은 tickets.md에 `DOG-` 접두 티켓으로 **기록만** 한다 —
  웨이브 편성은 판정 세션에서만.
- 판정의 갈래: (a) 마찰 해소 웨이브 (b) 차별 자산 심화(그래프 뷰·MCP 등) (c) 외부 검증 확대
  (d) 신호 없음 → 빌드 중단, 포지셔닝/문제 재정의.

## 중단 규칙 (무한 반복 차단)

1. **웨이브는 마일스톤 탈출 조건에 기여할 때만 편성한다.** "백로그에 남아 있어서"는 착수
   사유가 아니다.
2. **크로스체크발 신규 티켓은 자동 승격 금지.** CRITICAL/HIGH만 현 웨이브에 편입, 나머지는
   백로그에서 판정 세션을 기다린다.
3. **M2 이후 기능 웨이브 금지.** M3 판정 대기 후보(그래프 뷰 B-1 · #44 플로우 노드 ·
   와이어 실렌더·편집 · MCP · ASM-015 잔여 · ASM-021)는 도그푸딩 신호가 지목할 때만 부활한다.
   (2026-07-03 승격분: 패턴 프리미티브 정리→M1-D, #46 계약의 읽기 렌더→M1-C.)
