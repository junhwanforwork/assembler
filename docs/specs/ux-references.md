> 용도: ux-strategy.md의 비교 기준선. Assembler 북극성 = "스펙은 한 곳, 변경은 한 번" (1→N 변경 반영).

## 교차 비교

| 축 | claude.ai | NotebookLM | manyfast.io (Manifest) |
|---|---|---|---|
| 온보딩 (진입→첫 가치) | 3단계 — 프롬프트→자동 분할·아티팩트 | 3단계 — 노트북→소스 추가→인용 답변 | 5단계 — 가입→프로필→프로젝트→시작방식 3택→PRD·기능·플로우 순차 |
| 레이아웃 | 챗 1분할 → 산출물 발생 시 **자동 2분할** | **3분할 고정**(Sources/Chat/Studio) + 태스크별 2패널 접기 | 탭(문서타입) × 뷰(트리·디렉토리) 캔버스 + 우상단 Manny 챗 |
| AI 진입점 | 1주(챗)+인라인 "Edit with Claude"+스타터 칩 | 챗 + Studio 원클릭 버튼 다발 | 3+ — 홈 입력창·Manny 챗(편집 실행 권한)·질문지·업로드 |
| 산출물 구조 | 고립 아티팩트(상호 연결 없음) | 일회성 스냅샷(소스 읽기전용·단방향) | 생성 의존성 있는 문서 세트(PRD→기능→플로우, IA↔기능) — **변경 전파는 미확인** |
| 신뢰 장치 | 버전 셀렉터·Preview/Code | **인라인 인용 칩→원문 하이라이트 점프** | 변경 diff **개별/일괄 승인**(파란 화살표) |

**종합 시사점**: 3제품 모두 "생성의 순간" UX는 성숙(자동 분할·원클릭·승인 게이트)하나, **변경 1건이 연결 구조 전체에 전파되는 UX는 셋 다 부재.** Assembler의 챗+변경 계획 도크는 (claude 자동 분할)+(NotebookLM 인용 점프)+(Manifest 승인 게이트)를 합성하되 **전파 범위(영향받는 연결 산출물) 시각화**를 얹는 것이 유일 차별 자산.

## 1. claude.ai — 대화→행동 전환

- 산출물이 "한 줄보다 커지는 순간" **시스템이 자동으로 좌 챗/우 아티팩트 분할** — 사용자가 버튼을 찾지 않음.
- 아티팩트 위 하이라이트→"Edit with Claude"→그 자리 수정. 대화 분기=산출물 버전 분기. Preview/Code 토글.
- **차용**: ① 변경 계획이 생기는 순간 도크 자동 오픈 ② 하이라이트→인라인 AI 편집(스펙 블록 단위) ③ Preview/Code = 렌더뷰↔구조뷰 전환 ④ 대화 분기=버전 분기(이력·롤백 참고).
- **차별**: 그들은 대화가 원본·아티팩트는 고립 부산물 / 우리는 구조가 원본·챗은 조작 도구. 백지 시작 아닌 구조-컨텍스트 시작.
- 출처: support.claude.com/en/articles/9487310 · anthropic.com/news/projects · waboom.ai/blog/claude-artifacts-explained
- 미확인: 신규 챗 제안 칩 현행 라벨·개수, 분할 비율, 모바일.

## 2. NotebookLM — 3분할·그라운딩

- **Sources(입력)→Chat(합성)→Studio(출력) 3분할** — 파이프라인 위치를 레이아웃이 말해줌. 태스크별 2패널 접기(4구성).
- 모든 답변에 인용 칩 [1][2] — 호버=미리보기, **클릭=소스 패널 열리며 원문 하이라이트 점프**. 디자이너 증언: 인용은 부가기능이 아니라 신뢰의 핵심("massive user desire → quickly pivoted").
- content-first 온보딩: 백지가 아니라 "내 콘텐츠"로 시작.
- **차용**: ① 3분할 멘탈 모델(LeftRail↔Center↔Right와 동형 — 기준선) ② AI 제안의 근거 스펙 블록 점프 인용 칩(최우선) ③ 적응형 패널 접기 ④ content-first 온보딩(기존 기획 메모 먹이기).
- **차별**: 그들은 소스 읽기전용·출력 스냅샷·단방향 / 우리는 소스=산출물=살아있는 구조, 출력을 고치면 상류 되반영.
- 출처: jasonspielman.com/notebooklm(리드 디자이너) · support.google.com/notebooklm/answer/14276468 · blog.google notebooklm-new-features-december-2024
- 미확인: 소스 추가 직후 자동 제안 칩 현행, 소스 0개 빈 화면 문구, Studio 버튼 전체 목록.

## 3. manyfast.io(Manifest·에이전트 "Manny") — 직접 경쟁

- 한국어. "대화하세요, 기획이 완성됩니다". 온보딩 5단계, 시작 방식 3택(질문지/업로드 분석/빈 프로젝트). 생성 순서 고정: PRD→기능명세→유저플로우(+IA·와이어).
- 연결: **생성 시점 의존성**(PRD 있어야 플로우 생성, IA↔기능 연결) 존재. 단 **변경 시 자동 전파 근거는 공개 자료에 없음** — 북극성이 노리는 정확한 빈틈.
- Manny = 편집 실행 권한 보유(디자인 모드=합의 형성 / 실행 모드=즉시 수정). 변경 후 **파란 화살표→디렉토리 뷰 diff→개별/일괄 승인** = 우리 변경 계획 도크의 최근접 경쟁 구현.
- 내비 = 탭(문서타입) × 뷰(트리/디렉토리) 2층. Export = Excel·이미지·MD + **MCP(Claude Code·Cursor 직결)** — 우리 "구현 컨텍스트 내보내기"와 정면 충돌 영역.
- 과금: 크레딧(챗 응답도 소모, stop 환불 없음) — 탐색 위축 설계. 우리는 탐색(읽기)과 변경(쓰기) 비용 분리 여지.
- **차용**: ① 승인 UX 장단 기준선 ② 시작 3택 온보딩 ③ 탭×뷰 2층 내비 ④ MCP 내보내기 패리티 체크.
- **차별**: 캔버스 안 노드·엣지(문서 내부 표현) vs 우리의 산출물 간 참조 그래프 1급 시민. 변경 시점 전파.
- 출처: manyfast.io · docs.manyfast.io(/plan/user-flow.md·/plan/ia.md·/use/ai.md·/concept/interface.md) · blog.manyfast.io/playbook
- 미확인: 에디터 물리 패널 배치 전모(Manny가 상시 패널인지), PRD 변경 시 하류 자동 갱신 여부, 와이어 편집 상세.
- Playwright 캡처 권장: manyfast.io(히어로), editor.manyfast.io(게스트 진입 확인), docs.manyfast.io/concept/interface·/use/ai(도해 이미지).

### 4. 클로드 데스크톱 — 오버레이 문서 창

- 계획 문서가 **열고 닫는 임시 창**으로 뜬다 — 본 작업 화면을 대체하지 않는 비차단 참조 뷰.
- **레이어드 입체감**: 창이 뜰수록 배경 명도와 그림자가 함께 올라가 "위에 떠 있음"이 읽힌다.
- **차용**: ① `ui/OverlayPanel`(P-B) — 포털·백드롭·Esc·포커스 트랩·slide-in, side/window 변형 ② elevation 4단 짝 규칙(P-A, 명도·그림자 동반 상승). 기존 플로팅 라운드 방향(Stitch)과 정합.
- **차별**: 그들은 대화의 부산물 문서 / 우리는 문서가 구조의 투사(1급 산출물) — 창은 참조용 임시 뷰, 정본은 구조.
- 출처: 사용자 제공 스크린샷 2026-07-07.
- 미확인: 원 제품의 창 전환 애니메이션 상세·크기 조절 동작(스크린샷 밖).

### 5. git 상태 바 — 칩 나열 플로팅 바

- 화면 하단에 좌우로 긴 플로팅 바: **좌측 레포·브랜치 라벨, 우측 +N/−N 변경량 칩·"PR 생성" 버튼·CI 상태 칩.**
- 비개발자에게 git 상태를 **안전하게(읽기 문법으로) 보여주는** 표면 — 수치 칩의 +/−가 색(positive/negative)으로 즉시 읽힌다.
- **차용**: `ui/FloatBar`(P-D) — bottom-full 도킹 + 칩 문법(라벨 텍스트·수치 칩=Badge tone·상태 dot). 실소비는 아크2 v2(git 쓰기)에서, 이번엔 문법만 완결.
- **차별**: 그들은 개발자 도구의 상태 표시 / 우리는 비개발자 보호 장치 — "지금 무엇이 바뀌었고 다음에 무엇이 일어나는가"를 잇는 연결 표면.
- 출처: 사용자 제공 스크린샷 2026-07-07.
- 미확인: 원 제품의 칩 클릭 동작·CI 상태 전이 표현(스크린샷 밖).

### 6. 호버 해석 카드 — 장점/단점 구조화

- 카드에 마우스를 올리면 뜨는 구조화된 설명: **제목 → 장점(positive 색·불릿) / 단점·주의(negative 색·불릿)** (축구 인스윙어/아웃스윙어 예시).
- 설명 한 줄이 아니라 **트레이드오프가 보이는** 카드 — 선택지를 비교 가능한 형태로 준다.
- **차용**: `ui/InsightCard`(P-C) — 제목·AI 추정 배지·요약 + 좋은 점/주의할 점 불릿 섹션(Badge tone 재사용). v1.5 API 해석 AI의 출력 구조(요약+좋은 점+주의할 점)에도 반영.
- **차별**: 그들은 정적 지식 카드 / 우리는 사실(구조)과 추론(AI 해석)을 분리하고 환각을 차단하는 해석 계층(db-learning 규칙).
- 출처: 사용자 제공 스크린샷 2026-07-07.
