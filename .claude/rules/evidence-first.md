# Evidence First — 환각·헛짓 방지 행동 규율

모든 세션(오케스트레이터·레인·에이전트) 공통. 모델이 무엇이든(Opus·Fable·차기) 지켜야 하는
**행동 규칙**이다 — 똑똑함에 기대지 않고 구조가 실수를 막는다. diagnose-before-change의 강화판.

---

## 5계명

1. **주장에는 파일:라인.** "X가 Y한다"는 서술은 실제 코드를 열어 확인한 위치와 함께만 쓴다.
   확인 못 했으면 **"확인 필요"를 명시**하고 주장하지 않는다 — 추측을 사실 문장으로 쓰는 순간 환각이 된다.
2. **존재 확인 먼저.** 파일 경로·함수명·티켓 번호·스키마 컬럼·패키지 API를 인용하기 전에
   실물을 연다(Glob/Grep/Read·tickets.md·마이그레이션 파일). 기억·패턴으로 경로를 지어내지 않는다.
   (선례: 티켓 오기 사고 ASS-180 · 패킷 jsonb 오전제 사고 2026-07-08)
3. **같은 단계 2회 연속 실패 = 중단.** 같은 접근을 3번째 반복하지 않는다 — 멈추고
   ① 전제를 의심(진단으로 복귀) ② 실패 로그 원문을 다시 읽기 ③ 그래도 막히면 보고.
   반복 iteration은 대부분 잘못된 전제 위에서 돈다.
4. **검증 없는 "됐다" 금지.** 완료 선언 전 tsc·lint·테스트(해당 시 build·e2e)를 실행하고
   **마지막 줄 수치를 그대로** 기록한다. "통과할 것이다"는 보고가 아니다.
   테스트 green ≠ tsc green ≠ build green — 셋은 서로를 대신하지 못한다(2026-07-08 /s 플래그 사례).
5. **경계를 넘는 가정은 실물 대조.** 내 코드가 소비하는 상대(서버 산출물·타 레인 계약·DB 스키마·
   외부 응답 shape)는 생산자 쪽 실물을 확인해 맞춘다 — 내 구현에 맞춘 자작 픽스처는 가짜 green이다.

## 자기 점검 (완료 보고 직전 30초)

- [ ] 내가 인용한 경로·이름·번호를 전부 실물로 확인했나?
- [ ] "아마·~일 것"으로 쓴 문장이 결과 보고에 남아 있나? (있으면 확인하거나 "확인 필요"로)
- [ ] 검증 수치를 실행 출력에서 복사했나? (기억으로 쓰지 않았나)
- [ ] 받은 지시(패킷 원문·추가 지시·보충)를 **번호별로** 대조했나?

## 산출물 규칙

- 문서·주석에 수치·이름을 쓸 때 출처가 실행 출력/실파일이 아니면 쓰지 않는다.
- 요약(summary)·설명을 **지어내지 않는다** — 코드에서 확인 안 된 설명은 빈 값 + "후속(AI 해석) 몫" 표기.
