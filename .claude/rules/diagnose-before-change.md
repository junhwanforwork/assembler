# 변경 전 진단 — Diagnose Before Change

코드를 바꾸기 전에 "지금 어떻게 동작하는지"와 "무엇이 문제인지"를 먼저 확인한다.
추측으로 고치지 않는다. 모든 dev 작업(FE·BE·QA)의 기본 자세 — 가벼운 always-on 규율.

---

## Origin

perf 라운드에서 "근거 없이 최적화 금지" 7단계(`perf-diagnosis.md`)를 만들고 보니, 그 밑바탕
사고방식(증거→최소 변경→검증)은 성능에만 쓸 게 아니라 모든 변경에 맞았다. 단 풀 절차를 사소한
작업까지 강제하면 과하므로 *가벼운 공통 규율*로 분리하고, 성능·버그·고위험은 전용 절차로 위임한다.

---

## 4단계 (경량)

1. **진단** — 바꾸기 전 현재 동작·관련 코드·상태 흐름을 읽는다. 기존 함수/패턴 재사용 우선.
2. **증거** — 무엇이 왜 문제인지 파일:라인 또는 재현으로 짚는다. (No evidence, no fix.)
3. **최소 변경** — 가장 작은 범위로. 선제적 추상화·광역 리팩터 금지(YAGNI).
4. **검증** — tsc·lint·build 통과 + 동작 확인. 바꾼 것과 이유를 한 줄로.

리서치(웹 문서)는 **선택** — 원인이 비자명할 때만. 사소한 변경엔 생략.

---

## fast floor 예외

오탈자·1줄 텍스트·단일 토큰 값 교체는 이 절차 생략(CLAUDE.md fast floor와 동일).

## 에스컬레이션 (전용 절차로)

- 성능(느림 신호) → `perf-diagnosis.md` (증거→가설→타겟 리서치 7단계)
- 버그 → CLAUDE.md Bug Triage (QA 진단 → fix → QA 검증)
- 결제·보안·RLS·스키마 → `/multi-team` 병렬 3팀
