# Assembler UX Writing Rules (앱 UI 카피)

앱이 사용자에게 노출하는 텍스트에 적용. 버튼·에러·토스트·모달·빈 상태·안내 문구 전부 해당.
(생성되는 제품 객체 텍스트 문체는 `assembler/content-style.md` — 별개다.)

---

## 1. 기본 톤 & 문체

### 해요체 고정

- 모든 문구는 **해요체**로 통일한다
- 합쇼체(`~니다`), 반말(`~해`) 사용 금지

```
✅ "포인트가 적립됐어요"
❌ "포인트가 적립되었습니다"
❌ "포인트 적립됨"
```

### 능동형 우선

- 수동형(`~됐어요`, `~됩니다`)보다 능동형(`~했어요`) 우선
- `~었` 불필요하면 생략 (`됐어요` → `돼요`)

```
✅ "저장했어요"
❌ "저장됐어요"

✅ "인증이 완료돼요"
❌ "인증이 완료되어요"
```

**수동형 허용 예외:**

- 서비스 종료·기간 만료 (`서비스가 종료돼요`)
- 사용자에게 미치는 영향 안내 (`포인트가 차감돼요`)
- 민감 정보 수집 안심 문구 (`암호화되어 저장돼요`)

### 긍정형 우선

- 부정형(`없어요`, `안 돼요`)보다 긍정형 표현 우선

```
✅ "지금은 참여할 수 있는 폴이 없어요" → "새 폴이 곧 올라올 거예요"
✅ "잔액이 부족해요. 충전 후 이용해 보세요"
❌ "잔액이 부족합니다"
```

**부정형 허용 예외:**

- 정책상 명백히 불가능한 경우 (`미성년자는 출금할 수 없어요`)
- 일부 기능 미지원 명시 (`이 기기에서는 지원하지 않아요`)
- 사용자 선택 결과 제약 (`삭제하면 복구할 수 없어요`)

## 2. 경어 사용

### 과도한 존댓말 금지

| 금지          | 대체                   |
| ------------- | ---------------------- |
| `~시겠어요?`  | `~할까요?`             |
| `계시다`      | `있다`                 |
| `여쭤보다`    | `확인하다`, `물어보다` |
| `께`          | `에게`                 |
| `주시겠어요?` | `해 주세요`            |

```
✅ "연락처를 확인해 주세요"
❌ "연락처를 여쭤봐도 될까요?"

✅ "지금 앱을 사용하고 있나요?"
❌ "현재 앱을 이용 중이신가요?"
```

### 경어 허용 상황

- 사용자 상황 추정이 필요한 질문: `"혹시 계좌를 잘못 입력하셨나요?"`
- 협조 요청(설문, 리뷰): `"의견을 남겨주시면 더 좋은 서비스를 만들 수 있어요"`

## 3. 에러 메시지

### 핵심 원칙: Navigating Error

**에러 메시지의 목적은 "다음으로 나아가도록 안내"하는 것이다.**  
무엇이 잘못됐는지보다 어떻게 해결하는지를 먼저 알려준다.

### 에러 메시지 구조

```
[무슨 상황인지] + [어떻게 하면 되는지]
```

```
✅ "잠시 후 다시 시도해 주세요" (서버 오류)
✅ "네트워크 연결을 확인하고 다시 시도해 주세요"
✅ "계좌번호를 다시 확인해 주세요" (입력 오류)

❌ "오류가 발생했습니다" (원인·해결책 없음)
❌ "ERROR: 500 Internal Server Error" (기술 용어 노출)
❌ "잘못된 입력입니다" (무엇이 잘못됐는지 불명확)
```

### 에러 유형별 템플릿

**네트워크·서버 오류**

```
"일시적인 오류가 생겼어요. 잠시 후 다시 시도해 주세요."
```

**입력값 오류**

```
"[필드명]을 다시 확인해 주세요."
"[조건]으로 입력해 주세요."
```

**권한·자격 오류**

```
"[이유]로 이용할 수 없어요. [대안]해 주세요."
```

**서비스 점검**

```
"[서비스명]이 점검 중이에요. [시간]에 다시 이용해 주세요."
"언제 끝날지 확인 중이에요. 잠시만 기다려 주세요." (시간 미정)
```

### 회사 귀책 에러

- 사용자 탓으로 돌리지 않는다
- "죄송해요" 남용 금지 — 해결책 제시가 우선

```
✅ "지금 일시적으로 연결이 불안정해요. 잠시 후 다시 시도해 주세요."
❌ "서버 오류로 인해 처리할 수 없습니다. 죄송합니다."
```

## 4. 버튼 텍스트

### 기본 형태: `동사 + 하기`

```
✅ "확인하기", "저장하기", "삭제하기", "시작하기", "다시 시도하기"
❌ "확인", "저장", "삭제" (명사형 단독)
❌ "제출" (한자어 단독)
```

**예외 — 단독 "확인" 허용:**

- 단순 닫기 용도의 단일 버튼

### 다이얼로그 버튼 규칙

- 왼쪽(취소) 버튼: **"닫기"** 고정
  - "취소"는 작업 자체가 취소됐다는 오해 유발

```
✅ [닫기] [삭제하기]
❌ [취소] [삭제]
```

### 파괴적 액션 버튼

- 되돌릴 수 없는 동작은 구체적으로 명시

```
✅ "영구 삭제하기"
✅ "계정 탈퇴하기"
❌ "확인" (위험 동작에 모호한 버튼)
```

---

## 5. 빈 상태 (Empty State)

### 구조

```
[왜 비어있는지] + [어떻게 채울 수 있는지]
```

```
✅ "아직 참여한 폴이 없어요. 첫 번째 폴에 참여해 보세요."
✅ "만든 설문이 없어요. 새 설문을 만들어 보세요."
❌ "데이터가 없습니다"
❌ "No data"
```

---

## 6. 토스트 & 스낵바

- 짧게, 행동 결과만: `"저장했어요"`, `"복사했어요"`, `"삭제했어요"`
- 15자 이내 권장
- 해결 필요 상황이면 에러 메시지 패턴 사용 (토스트 아님)

---

## 7. 포인트·숫자 표기

- 포인트: `100P` (단위 붙여쓰기, 대문자 P)
- 금액: `10,000원` (천 단위 콤마)
- 날짜: `4월 12일`, `2026년 4월 12일` (숫자 + 한글 조사)
- 시간: `오전 9시`, `00시` (자정은 `0시` 아닌 `00시`)

---

## 8. 금지 표현 목록

| 금지                  | 이유           | 대체                   |
| --------------------- | -------------- | ---------------------- |
| `~입니다`             | 합쇼체         | `~이에요`, `~예요`     |
| `처리 중입니다`       | 딱딱함         | `처리하고 있어요`      |
| `완료되었습니다`      | 수동+합쇼체    | `완료됐어요`           |
| `불가합니다`          | 한자어 딱딱함  | `할 수 없어요`         |
| `재시도`              | 한자어 명사    | `다시 시도하기`        |
| `로딩 중`             | 기술 용어      | `불러오는 중이에요`    |
| `ERROR`, `404`, `500` | 기술 코드 노출 | 사용자 언어로 풀어쓰기 |
| `~의 경우`            | 번역투         | `~하면`, `~할 때`      |
| `~을 통해`            | 번역투         | `~로`, `~에서`         |
| `진행해 주세요`       | 모호함         | 구체적 동작 명시       |

---

## 9. assembler 서비스별 고정 문구

### 콘텐츠 (구현 스크랩)

- 검수 대기: `"검수 후 공개돼요"`
- 비공개 상태: `"아직 공개되지 않은 콘텐츠예요"`
- 저장 완료(비로그인): `"이 브라우저에 저장했어요"`
- 저장 완료(로그인): `"내 보관함에 저장했어요"`

### FEATURE.md (Phase 2)

- 결제 안내: `"FEATURE.md 1건에 ₩29,000이에요"`
- 생성 중: `"FEATURE.md를 만들고 있어요"`
- 다운로드 완료: `"FEATURE.md를 받았어요"`

### 공통

- 네트워크 오류: `"네트워크 연결을 확인하고 다시 시도해 주세요"`
- 서버 오류: `"일시적인 오류가 생겼어요. 잠시 후 다시 시도해 주세요"`
- 빈 보관함: `"아직 저장한 콘텐츠가 없어요. 마음에 드는 구현을 저장해 보세요"`
