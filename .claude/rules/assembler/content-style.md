# Assembler Content Style (생성 콘텐츠 문체)

## Origin

이건 **AI가 생성하는 제품 객체 텍스트**의 문체다 — 앱 자체 UI 카피(`ux-writing.md`의 해요체)와 **대상이 다르다**.
객체 설명을 마케팅 문장("사용자는 이 버튼을 눌러 가입할 수 있습니다")으로 쓰면 관계가 흐려지고
스펙으로서 쓸모가 없다. Assembler 객체는 **명사구 + 명시적 관계**로 적는다.

> 주의: `ux-writing.md`(해요체)는 **앱 UI 카피**용. 이 파일은 **생성되는 제품 객체** 내용용. 섞지 않는다.

---

## 원칙

- **명사구·직접 동작** 우선. 장황한 문장 금지.
- **관계를 명시**한다(Action / API / Database / Result).
- 피한다: 마케팅 표현, 불필요한 설명, 모호한 묘사.

## 예시

```
Bad:  "사용자는 이 버튼을 클릭하여 회원가입을 할 수 있습니다."

Good: Submit Button
      Action: Click
      API: POST /signup
      Result: Create User
```

## API / Database 설명

- 엔드포인트만 적지 말고 **목적(purpose)** 을 적는다.
- 누가 트리거하고(Triggered By), 어디서 쓰이고(Used In), 어떤 테이블을 건드리는지 명시.

## 불명확할 때

추정을 사실로 적지 않는다. 다음 중 하나로 표시:
- `정책 확인 필요` · `개발 확인 필요` · `QA 확인 필요`

## 언어

- 생성 콘텐츠 언어는 **입력(사용자 프롬프트) 언어를 따른다**(기본값). 객체 키/타입명은 영어 유지.
- (이 기본값은 열린 질문 — 사용자 확정 시 갱신.)
