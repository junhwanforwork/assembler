# /multi-team — 멀티팀 협업 (순차 릴레이 기본 + 고위험 병렬)

같은 티켓을 관점이 다른 3개 역할(A 실용 / B 안정 / C 구조)이 협업해 품질을 끌어올린다.
두 모드가 있고 **티켓 위험도로 자동 선택**한다.

| 모드                   | 흐름                                              | 비용                 | 언제          |
| ---------------------- | ------------------------------------------------- | -------------------- | ------------- |
| **순차 릴레이 (기본)** | A가 구현 → B·C가 관점별 리뷰·토론 → 구현자가 개선 | 단일 개발의 ~1.5~2배 | 대부분의 티켓 |
| **병렬 3팀 (opt-in)**  | 셋이 독립 구현 → 교차검증 → 심판이 베이스+패치    | ~3배                 | 고위험만      |

- 관점 정의: `.claude/rules/team-perspectives.md`
- 공유 보드: `docs/backlog/cross-review/OPIN-{id}.md` (`_template.md` 기반)
- `/cross-check`(단일 구현을 다중 리뷰어가 검증)와 구분 — multi-team은 구현 자체를 협업으로 만든다.

입력: 티켓 ID (`OPIN-110`) 또는 스펙 경로

$ARGUMENTS

---

## Step 0 — Fast floor (멀티팀 스킵)

아래는 멀티팀 비용이 아까운 변경이다. 멀티팀을 건너뛰고 단일 개발(필요 시 `/cross-check`)로 처리:

- 오탈자·1줄 텍스트/카피 수정
- 단일 상수·색상 토큰 값 교체
- 영향 파일 1개 & 로직 변경 없음

해당 시 "fast floor — 멀티팀 스킵" 출력 후 종료.

---

## Step 1 — 스펙 로드 + 모드 판정

1. 스펙 읽기: `docs/backlog/todo/OPIN-{id}.md` (없으면 `wip/`, `done/` 탐색)
2. 관련 정책/규칙 로드 (스펙의 `policy_refs`, `code_refs`)
3. `/tickets start OPIN-{id}` claim
4. 보드 생성: `_template.md` → `docs/backlog/cross-review/OPIN-{id}.md`, frontmatter(date, ticket_id, mode) 채우기
5. **모드 판정:**
   - **고위험 → 병렬 3팀 (Step P)**: 결제·포인트(billing/credits/point_ledger), 보안·인증, RLS/DB 스키마·마이그레이션 변경, priority `P1`, 또는 "접근 자체가 불확실"(설계 정답이 갈림)
   - **그 외 → 순차 릴레이 (Step 2~5)**

---

## 순차 릴레이 모드 (기본)

한 브랜치(worktree 1개)에서 진행한다. A가 짠 솔루션을 B·C 리뷰로 점진 개선 — 코드 일관성 유지, 비용 절감.

### Step 2 — A 구현 (Pragmatist)

`opin-fe` / `opin-be`를 **A 관점(실용·최소 변경)** 으로 호출해 티켓을 끝까지 구현한다.

빌드 게이트 (HARD STOP):

```bash
npx tsc --noEmit && npm run lint && npm run build
```

보드 §A에 초안 기록: 요약, `git diff --stat`, 트레이드오프, 자가 리스크.

### Step 3 — B 리뷰·토론 (Guardian)

B 관점으로 A의 diff를 **read-only** 리뷰한다: 엣지케이스·빈/에러 상태·권한·인증·정책·어뷰즈 누락을 C/H/M/L로 지적.

**토론 1라운드:** 각 지적에 구현자(A)가 _수용 / 반박_. 합의된 유효 지적만 채택한다(전부 반영 아님).
A가 채택 지적을 반영해 개선 → 빌드 게이트 재통과.
보드 §B에 지적 · 토론 결과 · 반영 내역 기록.

### Step 4 — C 리뷰·토론 (Architect)

C 관점으로 (개선된) diff를 리뷰: 토큰 하드코딩·SRP 위반·재사용 누락·네이밍을 지적.
토론 1라운드 → A 반영 → 빌드.
보드 §C 기록.

### Step 5 — 최종 + 적용

1. 최종 빌드 게이트 통과 확인.
2. 적용 (머지 또는 PR).
3. 보드 §최종 채우기, `/tickets verify OPIN-{id}` (브라우저/Playwright 검증까지 끝나면 `done`).

**에스컬레이션:** B 또는 C 리뷰가 _근본 접근을 갈아엎어야 할 수준_(설계 자체 재작성)이면 → 보드에 사유 기록 후 사용자에게 선택지 제시하고 대기. 단순 개선 범위면 그냥 반영.

---

## 병렬 3팀 모드 (고위험 전용) — Step P

서로 다른 접근을 독립적으로 만들어 비교할 가치가 있을 때만. worktree 3개 격리 필수.

```
P1. 3팀 동시 스폰 (team_name + isolation:"worktree"). 각자 독립 구현 + 빌드 게이트 → 보드 §1 기록.
    컨텍스트 공유 금지.
P2. 교차검증: 각 팀이 타 2팀 diff만 read-only 리뷰 → 보드 §2 매트릭스 (독립 판단).
P3. 심판 (code-reviewer): 점수표 → 베이스 1팀 선택 → 타팀 유효 지적을 패치로 정리.
    **합성 금지** — 베이스 1개 + 패치만. 막상막하/critical 충돌 → 보드에 에스컬레이션 후 사용자 대기.
P4. 베이스+패치 최종 빌드 → 적용. **패배 팀 worktree 정리** (git worktree remove). 보드 §최종 + 티켓 처리.
```

## 출력 요약 형식

```
## 멀티팀 결과 — OPIN-{id}  [모드: 순차 릴레이 / 병렬 3팀]

순차: A 초안 → B 지적 N(반영 M) → C 지적 N(반영 M)
병렬: 베이스 Team X ({관점}) + 패치 N (출처 {팀})
최종 빌드: tsc ✅ lint ✅ build ✅
보드: docs/backlog/cross-review/OPIN-{id}.md
티켓: verify / done

→ 또는 ⚠️ 에스컬레이션: {사용자 결정 필요 사유}
```
