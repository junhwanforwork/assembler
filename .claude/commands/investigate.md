버그를 체계적으로 조사한다. 근본 원인(root cause) 없이 수정하지 않는다.

**철칙: 증거 없이 고치지 않는다.**

## Phase 1 — 증상 수집

1. 버그를 정확히 서술한다 (어떤 상황, 어떤 입력, 어떤 결과)
2. 관련 파일/컴포넌트/API 라우트를 읽는다
3. 최근 변경사항 확인: `git log --oneline -20`
4. 재현 조건을 정리한다
5. **테스트 가능한 가설**을 하나 제시한다

## Phase 2 — 패턴 분석

알려진 버그 패턴과 대조한다:

- **레이스 컨디션**: 비동기 함수, 중복 요청, 상태 업데이트 순서
- **Null/undefined 전파**: optional chaining 누락, 타입 단언 오용
- **상태 오염**: zustand store가 의도치 않게 공유되는 경우
- **Supabase RLS**: 인증 컨텍스트 없이 서버 라우트에서 쿼리
- **Next.js 캐시**: 오래된 서버 컴포넌트 데이터
- **타입 불일치**: DB 스키마 vs TypeScript 타입 vs 실제 API 응답

git blame으로 누가, 언제 해당 코드를 변경했는지 확인한다.

## Phase 3 — 가설 검증

가설이 옳다면 어떤 증거가 있어야 하는지 먼저 정의한다:

```
가설: [...]
증거: 로그에서 [...] 가 보여야 함 / [...] 파일의 [...] 줄에서 [...] 값이어야 함
```

Bash 명령으로 로그를 확인하거나, 코드에서 값을 추적한다.

**3-Strike Rule**: 가설이 3번 틀리면 추측을 멈추고 사용자에게 에스컬레이션한다.

## Phase 4 — 수정

- 증상이 아닌 **근본 원인**을 수정한다
- 변경 범위를 최소화한다 (5개 파일 초과 시 사용자에게 알림)
- 수정 후 타입 체크: `npx tsc --noEmit`
- 빌드 확인: `npm run build`

## Phase 5 — 검증

- 원래 버그가 재현되지 않는지 확인
- 주변 기능에 regression 없는지 확인
- 디버그 리포트 출력:

```
## 디버그 리포트

### 근본 원인
[명확하게 한 문장으로]

### 수정 내용
[변경한 파일 + 이유]

### 상태
DONE / DONE_WITH_CONCERNS / BLOCKED
```

조사 대상: $ARGUMENTS
