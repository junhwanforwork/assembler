구현 시작 전 디자인 시스템·UX·정책·아키텍처 충돌을 점검한다.
플랜 수립 직후, 코드 수정 전에 반드시 실행.

## 목적

설계·정책과 충돌하는 구현을 사전에 차단한다.
❌ 항목이 하나라도 있으면 해결 후 구현 시작.

## 입력

$ARGUMENTS 에 티켓 ID 또는 작업 설명이 있으면 그 맥락으로 검사.
없으면 현재 In Progress 티켓 전체를 대상으로 검사.

## 검사 절차

### 1. 작업 범위 → 적용 룰 식별

수정 예정 파일의 도메인에 따라 체크할 룰 파일 결정:

| 경로                                                | 추가 적용 룰       |
| --------------------------------------------------- | ------------------ |
| `src/components/builder/**`, `src/app/(builder)/**` | `builder.md`       |
| `src/app/(main)/**`, `src/components/mypage/**`     | `ds-web.md`        |
| `src/app/api/**`, `supabase/**`                     | `api.md`           |
| `src/components/ui/**`                              | `ds-components.md` |

공통 적용 (모든 `.ts/.tsx`): `ds-tokens.md`, `ux-writing.md`, `button.md`, `file-structure.md`

### 2. 디자인 시스템

- [ ] 색상: `COLOR` 토큰 사용? 하드코딩 `#hex` 없음?
- [ ] 인터랙션: hover를 `INTERACTION` 토큰으로? (border 변경·box-shadow 강화 금지)
- [ ] 컴포넌트: `src/components/ui/`에 재사용 가능한 게 이미 있는가?
- [ ] 버튼: variant(solid/primary/neutral/danger/ghost)·size·텍스트(`동사 + 하기`) 규칙?
- [ ] 타이포그래피: weight 결정 트리 준수? TYPOGRAPHY 토큰 사용?
- [ ] semantic CSS class 네이밍: 주요 레이아웃 요소에 `_area`·`_wrap`·`_header`·`_body`·`_footer` 접미사?

### 3. UX Writing

- [ ] 해요체? (`~이에요`, `~예요` — `~입니다` 금지)
- [ ] 버튼 텍스트 `동사 + 하기` 형태?
- [ ] 에러 메시지: `[원인] + [해결 방법]` 구조? 기술 용어(`ERROR`, `500`) 노출 없음?
- [ ] 빈 상태: `[왜 비어있는지] + [어떻게 채울지]` 구조?
- [ ] 숫자 표기: 포인트 `100P`, 금액 `10,000원`, 날짜 `4월 12일` 형식?
- [ ] 금지 표현 없음? (`~입니다`, `처리 중입니다`, `완료되었습니다`, `로딩 중` 등)

### 4. 제품 정책

관련 파일만 선택하여 `docs/policy/` 에서 확인:

| 작업 도메인 | 확인 파일                 | 핵심 체크                                 |
| ----------- | ------------------------- | ----------------------------------------- |
| 포인트·출금 | `points.md`               | 최소 출금 10,000P, 수수료 8% (최소 500원) |
| 설문·빌더   | `survey.md`               | 상태 머신 draft→published→closed          |
| 마이페이지  | `mypage.md`               | 표시 정책                                 |
| 보상·심사   | `points.md` + `shared.md` | 보상 지급 플로우                          |
| 어뷰즈 방지 | `abuse.md`                | Rate limit, IP 중복                       |

### 5. 아키텍처

- [ ] 수정 파일 350줄 초과? → 먼저 분리 계획 수립
- [ ] 기존 훅·유틸 재사용 가능한 게 있는가?
- [ ] 새 파일 생성 시 단일 책임 원칙 충족?
- [ ] TypeScript strict 준수? `any` 사용 계획 없음?

## 출력 형식

```
## Pre-check: [티켓 ID / 작업명]

### 디자인 시스템
✅/⚠️/❌ 색상 토큰: ...
✅/⚠️/❌ 인터랙션: ...
(해당 항목만)

### UX Writing
✅/⚠️/❌ 해요체: ...
(해당 항목만)

### 정책
✅/⚠️/❌ [정책명]: ...
(해당 항목만)

### 아키텍처
✅/⚠️/❌ 파일 크기: ...
(해당 항목만)

### 결론
통과 → 구현 시작 가능
또는
수정 필요 → [구체적 조치 사항]
```

❌ 하나라도 있으면 해결 후 구현 시작.
⚠️ 는 구현 중 주의 사항으로 기록.
